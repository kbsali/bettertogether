Localization

Ruby came from Japan. Lua came from Brazil. Python came from the Netherlands.

Part of your responsibility as a major organization is to provide easy entry to services for speakers of any language. You will rarely hear that English is the lingua franca of programming from a non-native English speaker.

You wouldn't triumph a single programming language as a tool for every problem; similarly, you wouldn't hearld a single spoken language as the means for learning how to code, or use git, or share your ideas.

Support for CLAs

Recently, you introduced support for providing guidelines for contributions. This lowers the barrier for pull requests, and also ensures that every new submission to a repo is consistent, clean, and at a high standard.

You should provide a way for organizations to manage their Contributor License Agreement. Too often, CLAs are handled by users "signing" and emailing PDFs to a repo organizer, who then adds their name to a spreadsheet. Contributors are harrangued about whether "they've signed the CLA yet." Sometimes, submitters that send in a top-notch pull request drop off the face of the earth, leaving the code unmerged (and in a legal quagmire!).

Adminsitrators of organizations should be visually notified within the pull request whether the submitter has signed a CLA. The user making the pull request should know that as part of the pull request process, they're abiding by the CLA.

Labels for Notifications

I've complained about the old notification system for awhile. The new updates are simple, concise, and I like being able to keep track of every contribution and repo made by other members of my organization.

There's just one problem: any time I click on a notification, it's instantly marked as read, and tucked away into an "All notifications" black hole.

If I need to refer to a notificaiton I read, I can't really easily find it. I can't search through my notifications. I can't tag my notifications the same way I can tag my issues. Heck, if an issue is tagged, and I'm being notified about it, I should be able to at least see the tag in my Notifications Center without opening it up.

Notifications should be more like email. If I click on a notification, it shouldn't disappear. It should go away when I tell it to—sort of like archiving it. If I want to sort my notifications by labels, let me do so.

If you don't want to make notifications like email messages, that's totally fine. But please publicize the email address rules in a format outside of a blog post.

Rework the Network Graph

The Network Graph of forks is really awesome. I've used it several times to see what other contributions developers have made to a code base.

chjj's marked repo is a good example of the usefulness of the Network Graph. It's a JavaScript implementation of the GitHub-Flavored-Markdown. Let's say a user wants definition list support, similar to PHP Markdown Extra. But chjj doesn't want to implement that. That's completely fine; another user forks his project and implements the change. I can then browse other forks and decide which feature I want and need and use those implementations. It's a wonderful example of open-source development.

There are just a few usability issues with the graphs:

  1. Consider highlighting commits or forks that implement features the main repository does not want. For example, if #31 automatically links to a repo's issue #31, perhaps some other format could link to a forked repo (and highlight it on the original repo)
  2. Get rid of useless "+1" comments by adding a "thumbs up" counter. Allow for the ability to completely close threads; if the author doesn't want to implement the feature, it should be made abundently clear. Don't make the users do shit work by endlessly justifying their programming decisions or assuaging the frustrations of their userbase.
  3. Navigation on the Network Graph is a painful experience, the least problematic of which being that the up-and-down arrow keys often interfere with the browser's scroll bar.

Revamp Inline Ace Edits

You're using Ace to provide inline edits to files. Terrific! Ace is an incredible online code editor. (Disclaimer: I contribute to Ace.)

However, what is up with the horrendous white-on-black with minimal highlighting? The Ace editor is capable of so much more, on top of which, the color scheme is really unappealing. Why not provide the ability to set the themes, or improve the "Find" feature off of its default pop-up dialog mode?

For quick fixes, the inline editor is a wondeful tool, particularly in environments where git isn't installed on a machine (mobile phone, anyone?).

Discovering Nearby Projects

You present yourself as a social coding site, and your contributions to open-source development cannot be stressed enough.

Why not go the extra step and provide a map of nearby developers? You know which languages I like to work in most; you can also know my location (if I choose to tell you). Perhaps you should also suggest nearby hu-mans for collaborative hacking?

I'm not proposing you pivot into Facebook for Nerds. But if me and three other people have forked the same repository, and we've all made pull requests for it, it's be nice to have the opportunity to hash out some more ideas in person.

Autogenerate API Documentation

Documentation, you neglected necessity of open-source. How everyone marvels at your wonders, how few contribute to your success.

It's tremendously easy to generate wikis and GitHub pages. I'd like to see you also make it tremendously easy to create API documentation. Nothing too fancy; just click a button, parse my repository, and provide a list of public classes and members. If I, the developer, want to provide further improvements, then it's my responsibility to do so. But if I just don't have the time, it becomes incredibly useful for anyone stumbling around to get a list of what my code can do.

This becomes incredibly relevant for languages with members spread across various files.
